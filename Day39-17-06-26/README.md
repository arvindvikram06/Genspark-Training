## Concepts Explored

### 1. Multi-Stage Builds
Multi-stage builds allow you to use multiple `FROM` statements in your `Dockerfile`. 
- **Why it's used**: It keeps the final production image size small. The build stage uses a heavier SDK image (e.g., `mcr.microsoft.com/dotnet/sdk:10.0`) to compile the code. The final stage copies only the compiled artifacts into a much lighter runtime image (e.g., `mcr.microsoft.com/dotnet/aspnet:10.0`).

### 2. Docker Volumes
Volumes are the preferred mechanism for persisting data generated by and used by Docker containers.
- **Why it's used**: By default, data inside a container is lost when the container is deleted. Volumes mount a directory from the host machine into the container, ensuring that data like database records or uploaded files survive container restarts or deletions.

### 3. Docker Networking
Docker networking allows containers to communicate securely with each other, the host, or external networks. The lab explored how containers on the same bridge network can communicate using each other's container names.

## Key Commands Used

- `docker volume create pgdata`
  - **Explanation**: Creates a new, persistent Docker volume named `pgdata` that can be attached to containers.
- `docker volume ls`
  - **Explanation**: Lists all Docker volumes on the local system.
- `docker network create my-network`
  - **Explanation**: Creates a custom user-defined bridge network named `my-network`.
- `docker network ls`
  - **Explanation**: Lists all Docker networks available on the host.
- `docker run -d --name my-db --network my-network -v pgdata:/var/lib/postgresql/data postgres`
  - **Explanation**: Runs a PostgreSQL container named `my-db`, attaches it to the custom network `my-network`, and mounts the `pgdata` volume to persist the database data.
- `docker build -t libraryapi .`
  - **Explanation**: Triggers the multi-stage build process defined in the `Dockerfile` to create a lean production image.
